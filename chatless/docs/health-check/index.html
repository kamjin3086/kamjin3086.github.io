可用性检查（健康检查）

本文记录 Chatless 对“提供商是否可用”的设计思考与建议实现。

目标：判断“API 是否可达/可用（服务正常）”，而不是“密钥是否有效”。

设计动机

  • 仅测网络可达（如 ping、tcp 连通）不能代表 API 工作正常。
  • 用真实密钥发起调用会消耗额度/计费，不适合作为健康检查。

建议策略：带错误密钥的真实请求

  1. 构造一条最小可行的 API 请求（如 OpenAI 兼容接口 GET /v1/models 或一次最小输入的 POST /v1/chat/completions）。
  2. Authorization 设置为明确的假密钥（例如 Bearer sk-invalid）。
  3. 若返回 HTTP 4xx 且消息包含“未认证/密钥无效”等语义，即可判定：
    • 服务器可达
    • 路由已接收并处理
    • API 服务工作正常（只是凭证无效）
  4. 对于超时、连接失败、5xx 等，则判断为不可用或不稳定。

该策略避免了真实扣费/额度消耗，同时能覆盖 DNS、TLS、路由、网关、服务等链路的大多数问题。

通用示例（OpenAI 兼容接口）

curl -sS -X GET \
  -H "Authorization: Bearer sk-invalid" \
  -H "Content-Type: application/json" \
  https://api.openai.com/v1/models

期望收到 401/403 等认证失败响应；若长时间无响应或返回 5xx，则视为不可用/需稍后重试。

同理，可将 https://api.openai.com 替换为相应提供商或自托管网关的 Base URL 进行检查。

实现要点

  • 使用较短的超时时间与有限重试，避免阻塞 UI。
  • 控制频率（如冷却 1–5 分钟），避免触发速率限制。
  • 记录最近一次检查状态与时间戳，向用户展示“服务健康度”。
  • 在设置页暴露“检查端点”“超时/重试”等参数以便高级用户调整。

与真实调用的区分

  • 健康检查只回答“服务是否工作”。
  • 真正的业务请求由用户输入时触发，此时才使用真实密钥；密钥是否有效由业务响应决定。
